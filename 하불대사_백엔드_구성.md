# 하불대사 자동화 — 백엔드 구성도 & IT 체크리스트

> 목적: 하불(지급운임)을 **3원(三元) 대사** — ① 안전운임(계산 기준) ② 당사 내역 ③ 협력사 명세서 —
> 를 한 건마다 항목·총액으로 대조해 과오/미달/근거없는 청구를 자동 적발.
> 안전 원칙: **AI는 문서에서 추출만**, '정답(안전운임)'은 **기존 계산 엔진**이 산출,
> 불일치 건은 **사람이 검토**. (AI가 금액을 판단하지 않으므로 정확성 유지)

## 단계 (Phase)
- **1단계 — 대사**: 3원 대조로 이상(과다/과소/안전운임 미달/근거없는청구) 적발 → 대사표 ✅ PoC 완료
- **2단계 — 통보 메일**: 대사 결과를 관계자에게 메일 발송
  - **배차담당자(내부)**: 이상 건 요약 + 차액·사유 → 검토/조치 요청
  - **협력사(외부)**: 과다·근거없는 청구 건만 "금액 확인 요청" → **단, 발송 전 배차담당자 승인(초안→승인→발송)**
  - 주의: 협력사 메일은 외부·민감 → **자동 발송 금지, 사람 승인 게이트 필수** (안전운임 미달은 당사 내부 시정 대상이라 협력사 통보 X)
  - 기술: 백엔드 메일(SMTP/메일API)로 발송(첨부=대사표). 현 웹의 EmailJS(클라이언트)는 부적합

## 0. 대사 모델 (3원 대사)
안전운임 외에 **대기료·셔틀·부대비 등 추가비용**이 있으므로 단순 2자 비교가 아님.

| 항목 | ① 안전운임(계산) | ② 당사 내역 | ③ 협력사 명세서 | 판정 포인트 |
|---|---|---|---|---|
| 기본운임 | 계산값(하한) | 기록값 | 청구값 | ②③ 일치? ②≥① 준수? |
| 할증 | 계산값 | 기록값 | 청구값 | ②③ 일치? |
| 대기료/부대비 | (고시 기준 참고) | 기록값 | 청구값 | ②③ 일치? 근거 있나? |
| **합계** | 하한 | 지급기준 | 청구액 | 종합 판정 |

**대사 규칙(핵심 3가지)**
1. **협력사 명세서 ↔ 당사 내역**: 일치해야 정상. 불일치 = 과다/과소 청구 → 차액·항목 표시
2. **당사 내역 ≥ 안전운임**: 안전(위탁)운임 미달 불가(법적). 미달 시 **준수 위반 경고**
3. **추가비용 항목**: 협력사가 청구했으나 당사 내역에 없으면 → **근거 확인 필요** 플래그
   (반대로 당사 인정 항목이 명세서에 누락되면 누락 표시)

## 1. 전체 데이터 흐름 (구성도)

```
 [정산 담당자]
     │  ① 거래명세서/하불내역 업로드 (PDF·스캔·엑셀)
     │     또는 사내 시스템에서 자동 수집
     ▼
 ┌─────────────────────── 백엔드 API 서버 ───────────────────────┐
 │ ② 문서 파싱 / AI 추출 (Claude 비전)                            │
 │     → 줄 항목 구조화:                                          │
 │       목적지 · 기점(항구) · 컨규격(20/40/45) · 운송유형(왕복/편도) │
 │       · 할증(냉동/위험물/야간 등) · 청구금액                    │
 │ ③ 정규화/매칭: 목적지→행정동(기존 FLAT 재사용), 항구명 표준화    │
 │ ④ ★ 안전운임 계산 엔진(기존 로직 재사용) → 안전운임 기준금액 산출 │
 │ ④' 당사 내역 수집(사내 시스템/업로드) — 안전운임+합의 추가비용     │
 │ ⑤ 3원 대사: ①안전운임 vs ②당사 vs ③협력사 → 항목·총액 비교        │
 │     판정: 과다/과소/안전운임 미달/근거없는 추가청구 + 차액         │
 └───────────────────────────────────────────────────────────────┘
     │                                   │
     ▼                                   ▼
 [DB] 명세서·항목·대사결과·이력 저장   [결과 화면 / 리포트]
                                       · 대사표(불일치 하이라이트)
                                       · 엑셀·PDF 내보내기
     │
     └─(선택) 사내 정산/오더/하불 시스템 연동
            : 하불 데이터 자동 수집 / 대사결과 회신
```

핵심: ④의 **계산 엔진은 현재 조회 앱에 이미 구현됨** → 서버로 이식/공유해 그대로 재사용.

## 2. 필요 모듈
1. **인증/권한** — 로그인, 정산담당 권한(사내 SSO 연동 가능 시 활용)
2. **문서 업로드·저장** — 파일 스토리지
3. **AI 추출** — Claude 비전 API 호출 (API 키는 **서버에만** 보관)
4. **정규화/매칭** — 목적지→행정동, 항구명 표준화 (기존 데이터 재사용)
5. **운임 계산 엔진** — 안전운임+할증 '정답 금액' (기존 로직 이식)
6. **대사 엔진** — 비교·차액·판정(허용오차·반올림 규칙 포함)
7. **DB** — 명세서/항목/대사결과/이력
8. **리포트/내보내기** — 엑셀·PDF, 불일치 요약
9. **(선택) 사내 연동 어댑터** — 하불/오더/정산 API
10. **로깅/감사 추적** — 금액 취급이므로 누가·언제·무엇을 대사했는지 기록

## 3. 실행 환경 옵션 (가볍게 → 무겁게)
- **A. 경량 서버리스**(클라우드 함수 + 매니지드 DB) — 빠른 PoC, 운영부담 적음
- **B. 사내 서버 / 사내 클라우드** — 내부망 연동·금융성 데이터 보안 강할 때
- 선택 기준: 데이터 민감도 · 사내 보안정책 · 연동 대상(사내 시스템) 위치

## 4. IT 체크리스트 (세방 IT에 확인할 것)

### A. 데이터 접근 / 연동 (3원 대사용 3개 소스)
- [ ] **① 안전운임**: 운송 조건(목적지·규격·운송유형·할증)을 어디서 확보? (오더/배차 데이터)
- [ ] **② 당사 내역**(세방 인정 운임+추가비용)은 **어느 시스템**에 있나? 추출 방법(API/DB/엑셀)?
- [ ] **③ 협력사 명세서**는 어떤 형태로 들어오나? (PDF / 스캔 / 엑셀 / EDI)
- [ ] **추가비용 항목 정의**(대기료·셔틀·부대비 등 기준·코드)는 어디에 관리되나?
- [ ] 하불(지급) 내역과 오더/정산 데이터가 키로 연결되나? (건별 매칭 가능 여부)

### B. 인프라 / 실행 환경
- [ ] 백엔드를 **어디서 돌릴 수 있나?** (사내 서버 / 사내 클라우드 / 외부 클라우드 허용?)
- [ ] **외부 AI API 호출**(클라우드 LLM)이 보안정책상 허용되나? (불가 시 사내망 내 모델 검토)
- [ ] 인터넷 아웃바운드 / 방화벽 정책

### C. 보안 / 계정
- [ ] 사용자 인증 체계 — 사내 SSO/AD 연동 가능?
- [ ] 금액·정산 데이터 취급 규정 — 저장 위치, 암호화, 보관기간
- [ ] 개인정보/거래정보 처리 정책

### D. 운영 / 책임
- [ ] 담당 부서 / **연동 창구**(누구와 협의?)
- [ ] 원천 데이터 정합성 책임(원본 오류 시 처리)
- [ ] 대사 결과 활용 범위 — **참고용** vs **정산 반영**

### E. 범위 / 규모
- [ ] 월 대사 건수(명세서 수 · 줄 수) → 처리량 산정
- [ ] 협력사 수 · 명세서 **양식 다양성**

## 5. 시작 제안 (PoC 먼저)
1. **사내 연동 없이** 샘플 거래명세서 몇 건을 **수동 업로드** →
   AI 추출 → 계산엔진 대사 → **결과표**까지만 먼저 구현
2. 추출 정확도 / 적발 가치 검증
3. 검증되면 **사내 연동(B단계)** 으로 확대 (하불 자동 수집 / 결과 회신)

> PoC는 외부 데이터 의존이 적어 빠르게 가치 증명 가능. IT 협조는 ④ 사내 연동 단계에서 본격 필요.
